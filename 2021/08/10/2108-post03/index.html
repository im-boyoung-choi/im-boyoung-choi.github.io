hello, JavaScript!

처음 블로그를 만든 목적은 패스트캠퍼스 ‘프론트엔드 개발 올인원 패키지’ 강의 중 JavaScript 파트인 ‘벨로퍼트와 함께하는 모던 자바스크립트’를 수강하며 개발 공부를 하는 것을 기록하기 위함이었다.

벨로퍼트 님의 GitBook을 참고해서 ‘1장. 자바스크립트 입문’의 마지막 파트를 공부하던 중 객체를 너무 겉핥기로 공부했다는 것을 알게됐다. 추가로 객체에 대해 공부하다 보니 벨로퍼트 님의 강의 문서만 따라해서는 초보자가 JavsScript를 쉽게 이해하기는 어려울 수 있다는 생각이 들었고, 작성했던 11개의 게시글을 지우고 다시 처음부터 정리를 시작한다.

벨로퍼트 님의 강의 문서를 토대로 작성하려고 했던 처음 목표와는 달리 나는 좀 더 초보자를 위해 TCP School 문서를 토대로 개념을 정리해가려 한다. 벨로퍼트 님의 강의에서 다루는 내용이 빠지는 것은 아니니 2개의 문서를 함께 참고하길 바란다.

1. JavaScript 개요

JavaScript는 객체 기반의 스크립트 언어다.

HTML로는 웹의 내용을 작성하고, CSS로는 웹을 디자인하며, JavaScript로는 웹의 동작(HTML 내용, 속성, 스타일 변경 등)을 구현할 수 있다. 주로 웹 브라우저에서 사용되지만 Node.js와 같은 프레임워크를 사용하면 서버 측 프로그래밍에서도 사용할 수 있다.

JavaScript는 1995년 넷스케이프(NetScape)의 브렌던 아이크(Brendan Eich)에 의해 만들어졌다. 처음에는 모카(Mocha)라는 이름으로 개발되었으나, 그 후 라이브스크립트(LiveScript), 최종적으로 자바스크립트(JavaScript)라는 이름으로 변경됐다.

JavaScript는 아래와 같은 언어적 특징을 가지고 있다.

  1. 객체 기반의 스크립트 언어
  2. 동적이며, 타입을 명시할 필요가 없는 인터프리터(interpreter, 해석기) 언어
  3. 객체 지향형 프로그래밍과 함수 지향형 프로그래밍을 모두 표현할 수 있는 언어

인터프리터(interpreter, 해석기) 언어와 컴파일(compile, 옮김) 언어

인터프리터 언어는 원시 코드(프로그래머가 작성한 소스 코드)를 기계어로 변환하는 과정(컴파일) 없이 해석하여 바로 명령어를 실행하는 언어이다. 소스 코드를 읽으며 실행하기 때문에 프로그램의 실행 시간은 느리다. 스크립트 언어라고도 불리며 대표적인 언어 예시로는 Python, R, Rubu 등이 있다.

컴파일 언어는 원시코드를 모두 기계어로 변환한 후 기계어를 CPU/메모리를 통해 읽는 방식으로 실행하는 언어이다. 컴파일하기 때문에 규모가 큰 프로그램이라면 컴파일 시간이 오래 걸릴 수 있지만, 컴파일 후 기계어를 통해 프로그램을 실행하기 때문에 실행시간은 빠르다. 대표적인 언어 예시로는 C++, Java 등이 있다.

(참고 사이트)

1996년 넷스케이프(Netscape)는 자바스크립트를 국제 표준안으로 만들기 위해 ECMA(European Computer Manufacturers Association)에 제출한다. 그 결과 ECMA는 ECMAScript라는 새로운 표준을 제정했고, 첫 번째 버전인 ECMA-262이 1997년에 공표된다.

ECMAScript는 JavaScript 뿐만 아니라 마이크로소프트의 JScript나 어도비의 ActionScript도 따르는 국제 표준이 된다. 현재 JavaScript의 최신 표준은 2015년에 발표된 ECMAScript 6(ES6, ES2015)이다.

JavaScript 최신 표준에 대한 정보는 ESMA 홈페이지에서, JavaScript에 대한 더 자세한 설명은 모질라 개발자 사이트(MDN)에서 살펴보자.

2. JavaScript 문법

프로그램(program)은 컴퓨터가 실행할 수 있는 명령(instruction)으로 이루어진다. 컴퓨터 프로그래밍에서 컴퓨터가 실행할 수 있는 명령들을 실행문(statement)이라고 한다. 즉, 프로그램이란 특정 결과를 얻기 위해서 컴퓨터에 의해 실행되는 실행문의 집합이라고 할 수 있다.

JavaScript의 실행문은 세미콜론(;)을 붙여 구분하는 것이 기본이다. 세미콜론을 붙이지 않더라도 인터프리트 과정에서 구문 오류가 발생하지는 않는다. 인터프리터가 문장의 끝이라고 생각되는 지점에서 자동으로 세미콜론을 붙여주기 때문이다. 이 기능을 세미콜론 자동 삽입이라고 한다.

이전에는 반드시 세미콜론을 넣자는 것이 압도적인 의견이었다면, 최근에는 세미콜론을 굳이 넣지 말자는 주장에도 힘이 실리고 있다. 여전한 논쟁거리지만 각자의 스타일대로 코드를 작성하면 될 것 같다. 나는 세미콜론을 넣는 형태를 사용할 예정이다.

JavaScript에서는 대소문자를 구분한다. 변수나 함수의 이름, 예약어 등을 작성하거나 사용할 때는 대소문자를 정확히 구분해서 사용해야 한다.

리터럴(literal)과 식별자(identifier)

리터럴(literal)은 12, "JavaScript", '안녕하세요', true 등과 같이 직접 표현되는 값 그 자체를 의미한다.

식별자(identifier)는 변수나 함수의 이름을 작성할 때 사용하는 이름을 의미한다. JavaScript에서 식별자는 영문자(대소문자), 숫자, 언더스코어(_), 달러($)만을 사용할 수 있다. 식별자 작성 방식은 Camel Case 방식이나 Underscore Case 방식을 사용한다.

Camel Case 방식과 Underscore Case 방식

Camel Case 방식은 식별자가 여러 단어로 이루어질 경우 첫 번째 단어는 모두 소문자로 작성, 그 다음 단어부터는 첫 문자만 대문자로 작성하는 방식이다. 관행적으로 Camel Case 방식을 더 많이 사용한다.

Underscore Case 방식은 식별자를 이루는 단어들을 소문자로만 작성하고, 그 단어들을 언더스코어(_)로 연결하는 방식이다.

키워드(keyword)

JavaScript에서 특별한 용도로 사용하기 위해 미리 예약해둔 단어들을 키워드(keyword) 또는 예약어(reserved word)라고 한다.

대표적인 키워드는 let, const, var와 같은 변수 정의 키워드, function과 같은 함수정의 키워드 등이 있다.

3. JavaScript 출력

JavaScript는 여러 방법을 통해 결과물을 HTML 페이지에 출력할 수 있다.

window.alert() 메소드

가장 간단하게 데이터를 출력할 수 있는 방법이다. window.alert() 메소드는 별도의 대화 상자를 띄워 사용자에게 데이터를 전달한다.

HTML DOM 요소를 이용한 innerHTML 프로퍼티

출력을 위해 가장 많이 사용되는 방법은 HTML DOM 요소를 이용한 innerHTML 프로퍼티를 이용하는 방법이다.

document 객체의 getElementByID(), getElementsByTagName() 등의 메소드를 사용하여 HTML 요소를 선택한 뒤 innerHTML 프로퍼티를 이용하면 선택된 HTML 요소의 내용이나 속성 값 등을 쉽게 변경할 수 있다.

document.write() 메소드

document.write() 메소드는 웹 페이지가 로딩될 때 실행되면서 가장 먼저 데이터를 출력한다. 그래서 대부분 테스트나 디버깅을 위해 사용된다.

웹 페이지의 모든 내용이 로딩 된 후에 document.write() 메소드가 실행되면, 먼저 로딩된 모든 데이터를 지우고 자신의 데이터를 출력하기 때문에 테스트 이외의 용도에서 사용할 때에는 충분한 주의가 필요하다.

console.log() 메소드

앞으로 내가 제일 많이 사용할 방법이다. console.log() 메소드는 웹 브라우저의 콘솔을 통해 데이터를 출력한다.

크롬의 Console 탭을 활용하기에는 코드 작성에 번거로움이 있어 GitHub에서 만든 Online Code Editor CodeSandbox에서 코드를 작성해볼 예정이다.

CodeSandbox에 접속해 로그인한 뒤 ‘Create Sandbox’를 눌러 Vanilla 템플릿을 열어준다. Vanilla 템플릿을 선택한 이유는 다른 라이브러리를 사용하지 않고 JavaScript만 사용할 예정이기 때문이다.

템플릿이 나타나면 오른쪽 상단에 ‘Fork’ 버튼을 누른다. 개발에서 Fork란 하나의 소프트웨어 소스 코드를 통째로 복사하여 독립적인 새로운 소프트웨어를 개발하는 것을 의미한다. (사실 여기서 ‘Fork’ 버튼을 눌러야 하는 이유는 잘 모르겠다. 아시는 분 댓글 부탁드립니다.)

조력자 님께서 댓글로 해당 부분에 대해 부가 설명을 해주셨다. (감사합니다!)

‘위에서 잘 설명해 주셨듯이 fork는 다른 사람의 원본 repository를 유지하고 원본 repository를 내 github에 독립적인 repository로 사용하기 위해 복제하는 기능입니다. fork를 하게 되면 원본 repository와 복제된 내 repository가 연결되어 있어 원본 repository에 변화가 생기면 forked된 repository(내 repository)에 반영할 수 있습니다.’

댓글을 읽어보니 확실히 이해가 됐다. GitHub Blog 테마를 적용할 때도 사용한 방법인데 fork의 개념만 쫓다 보니 어렵게 느껴졌나 보다. 다시 한번 쉬운 설명 감사합니다.
